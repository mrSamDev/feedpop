RSS Feed Reader — Assessment



Overview



Build a functional RSS feed reader. You've been provided with a static UI scaffold — your job is to bring it to life.



What's Provided



You have three files to start with:



- components.tsx — Presentational components for the feed list, articles, and modal. You may modify or extend them as needed.

- App.tsx — A layout with a sidebar and main content area, wired up with hardcoded sample data.

- styles.css — Styling for all provided components.



Run the app and click around to understand what's there before writing any code.



What's Expected



A user should be able to add RSS feeds by URL and read their articles. That's the core — make it work well.



The app should feel complete, not like a prototype. Real feeds from the internet should work. Errors should be handled in a way that helps the user, not confuses them. The UI should remain stable and responsive as content loads.



What We're Looking For



We're evaluating this the way we'd evaluate production code from a senior engineer:



- Does it work? Not just the happy path — what happens when things go wrong, when input is unexpected, when feeds behave differently from each other?

- Is it well-built? Clean architecture, sensible separation of concerns, appropriate use of TypeScript. Code that another developer can pick up and extend.

- Does it feel like a real product? Thoughtful UX decisions, attention to detail, and evidence that you considered what a real user would experience — not just what the requirements say.



Go Above and Beyond



The baseline gets you through the door. What makes this stand out is everything else. Think about what would make this a feed reader you'd actually want to use, and build that.



Show us your best work.



Guidelines



- You may install any libraries you need.

- You may use any tools you normally use, including AI assistants. If you do, we expect you to review, understand, and refine the output — treat it as a starting point, not a final answer. Code that looks generated and unreviewed (inconsistent patterns, generic error handling, unnecessary comments, dead code) will be noticed.

- You may restructure the provided code however you see fit — the scaffold is a starting point, not a constraint.
